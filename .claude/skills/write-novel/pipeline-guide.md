# 流水线补充指南（增强版：融合Superpowers工程化模式）

## 全局写作禁令

以下规则适用于所有Agent的所有输出：

1. **绝对禁止使用破折号（——）**：这是AI写作的最典型坏习惯。用逗号、句号、冒号或重组句子替代。编辑Agent必须将破折号标记为🔴严重问题。
2. **人物首次出场必须有外貌细节描写**：包括主角本人。不需要长篇大论，但要有具体视觉印象。
3. **逻辑合理性**：学生不可能工作日全天送外卖，需要交代时间背景（周末、假期等）。
4. **去AI味强制规范**（详见 `references/style-guide.md` "去AI味指南"章节）：
   - **词汇层面**：禁用AI高频词汇黑名单（首先/其次/此外、值得注意的是、微微/缓缓/淡淡超限、不禁/竟然超限等）。文体家写作时必须逐一对照检查。
   - **句式层面**：禁止连续3个以上等长句子、禁止连续两段同句式开头、每章必须有极短句和口语省略句。
   - **描写层面**：心理活动必须外化为动作/感官，每场景至少1个反逻辑日常碎片，每章至少1处感官暗示。
   - **对话层面**：角色语言风格必须有明显差异，连续纯对话不超过4句须穿插动作，允许打断和跑题。
   - **结构层面**：段落长度参差不齐，信息分布不均匀，章末钩子形式多样化（禁止每章都用"然而他不知道……"式结尾）。
   - 编辑Agent在"文字质量"维度中必须专项检查AI味并量化评级（🔴🟡🟢）。

## 核心变革：从"感觉驱动"到"证据驱动"

本流水线融合了 Superpowers 软件工程工作流的核心理念：

| Superpowers 概念 | 小说写作等价物 |
|---|---|
| TDD（先写测试再写代码） | 先写读者期望规格，再写正文 |
| Spec Compliance Review | 蓝图合规审查：剧本是否忠实执行了蓝图？ |
| Verification Before Completion | 证据驱动验证：每个判断都要引用正文原文 |
| Systematic Debugging | 根因分析：低分时诊断问题出在哪个环节 |
| Circuit Breaker | 迭代熔断：3次失败后强制决策 |

## 新增Agent：蓝图合规审查

Agent 指令：`.claude/agents/spec-reviewer.md`

在编剧产出后、文体家动笔前，验证剧本是否忠实执行蓝图：

```
编剧产出script → 蓝图合规审查 → ✅通过 → 文体家
                              → ⚠️偏差 → 作者决定(保留/修正)
                              → ❌偏离 → 回编剧
```

## 新增产出：读者期望规格（TDD类比）

架构师在产出蓝图后，同时产出读者期望规格：

```
蓝图完成 → 生成 reader-spec.md → 展示给作者确认
                                   ↓
                    编剧参考 → 文体家参考
                                   ↓
                    编辑验证 → 读者验证（运行测试）
```

## 研究员调用时机

研究员不在主流水线中，而是按需调用：

```
任何Agent执行中 ──需要真实信息──→ 暂停 → 调用研究员 → 获得结果 → 继续执行
作者brainstorm中 ──"查一下XXX"──→ 调用研究员 → 展示结果 → 继续讨论
独立调用 ──/write-novel research──→ 研究员 → 输出备忘录
```

## Brainstorm 交互细节

### 动态提问核心原则（适用于所有Agent）

**所有brainstorm的问题必须动态生成，禁止使用预设的固定问题清单。**

- 每轮问题必须基于作者上一轮的具体回答自然延伸
- 像资深编辑与作者的真实对话——敏锐捕捉亮点和盲区，顺势深挖
- 发现亮点要追问如何放大，发现薄弱点要追问如何补强，发现矛盾要指出并请作者取舍
- 只规定主题方向和最低问题数量，具体问题由Agent根据上下文动态决定

### 关键创作节点（新书启动：架构师 init）— 5轮深度交互

新书启动涉及大纲设计、人物肖像构建等根基性工作，**每轮动态生成至少5个有深度的问题**：

| 轮次 | 主题方向 | 目标 |
|------|---------|------|
| 第1轮 | 故事基因探索 | 深挖核心创意、冲突、读者体验、独特卖点 |
| 第2轮 | 世界观与金手指 | 机制细化、限制设计、成长路线、合理性验证 |
| 第3轮 | 人物肖像深度构建 | 立体人物、关系网、弧线、标志性特征 |
| 第4轮 | 大纲结构与节奏 | 30章规划、伏笔时间线、爽点分布、卷间递进 |
| 第5轮 | 开篇策略定稿 | 前3章规划、全局一致性、查漏补缺、最终确认 |

每轮展示给作者时使用以下格式：

```
════ 架构师 · 第X轮：[主题名] ══════

[本轮核心内容展示]

──── 深入讨论 ────

❓ [动态生成的问题1]
❓ [动态生成的问题2]
❓ [动态生成的问题3]
...（至少5个）

════════════════════════════════════

💬 请逐一回答以上问题，或提出你的想法。
• 说"查一下XXX"可调用研究员
• 如果对本轮内容满意，说"下一轮"进入下一主题
```

### 章节蓝图（架构师 chapter）— 5轮交互

每轮动态生成至少3个针对性问题：

| 轮次 | 主题方向 | 目标 |
|------|---------|------|
| 第1轮 | 上下文回顾与定位 | 前章衔接、本章任务、大纲调整 |
| 第2轮 | 核心事件设计 | 因果链、角色反应、情绪效果、后续影响 |
| 第3轮 | 场景与人物动线 | 场景划分、情绪变化、精简判断 |
| 第4轮 | 伏笔与钩子 | 伏笔操作、章末钩子力度、前文伏笔利用 |
| 第5轮 | 蓝图定稿 | 整体节奏确认、最终调整 |

### 编剧/文体家 — 5轮交互

每轮动态生成至少3个针对性问题：

| 轮次 | 编剧 | 文体家 |
|------|------|--------|
| 第1轮 | 场景拆分方案展示 | 正文初稿展示 |
| 第2轮 | 冲突节拍讨论 | 开篇节奏与代入感讨论 |
| 第3轮 | 对话方向与信息流 | 对话自然度与角色声音 |
| 第4轮 | 节奏与情绪曲线 | 描写密度与爽点呈现 |
| 第5轮 | 剧本定稿 | 章末钩子与正文定稿 |

每轮展示格式：

```
──── [Agent名]·第X轮：[主题] ────

[本轮内容]

❓ [动态生成的问题1]
❓ [动态生成的问题2]
❓ [动态生成的问题3]
...（至少3个）

────────────────────

💬 请审阅以上内容。你可以：
• 回答问题 / 提出修改意见
• 补充想法
• 说"继续"跳到下一轮
• 说"查一下XXX"调用研究员
```

### 加速规则

如果作者在某轮说"继续"/"OK"/"没问题"，可以合并后续轮次加速推进，但以下轮次**不可跳过**：
- **init**：前3轮（故事基因、世界观、人物肖像）
- **chapter**：前2轮（上下文回顾、核心事件）
- **编剧/文体家**：前2轮

## 章节定稿收尾清单（证据驱动版）

当一章通过审阅后，执行以下步骤。**每步必须有证据确认，不接受"应该没问题"。**

### Phase 1：完成验证（Verification Before Completion）
1. ☐ **蓝图核心事件覆盖**：逐条列出蓝图核心事件 → 引用正文对应段落
2. ☐ **读者期望规格达成**：引用读者验证报告中的达成记录
3. ☐ **人物一致性**：引用编辑报告中的一致性检查结果
4. ☐ **字数范围**：实际字数 = X（在2000-4000范围内）

### Phase 2：文件操作
5. ☐ 将 `drafts/chapter-{N}-draft.md` 复制到 `chapters/chapter-{NNN}.md`

### Phase 3：状态更新
6. ☐ 架构师更新 `story-state.md`：
   - 添加本章事件到时间线
   - 更新人物位置/状态
   - 更新伏笔状态
   - 记录章末钩子
   - 更新章节计数和总字数
7. ☐ 架构师更新 `characters.md`：
   - 新人物添加条目
   - 关系变化更新
   - 心理状态更新
8. ☐ 架构师更新 `outline.md`：
   - 标记章节为"定稿"
   - 记录偏差（如有）
   - 记录新灵感（如有）
   - 记录迭代历史（如有多次修改）

### Phase 4：状态验证（二次确认）
9. ☐ **读取更新后的 `story-state.md`**：确认时间线与正文一致
10. ☐ **读取更新后的 `characters.md`**：确认人物状态与正文末尾一致
11. ☐ 向作者报告更新内容

## 根因分析参考表

追读分不通过时，使用以下表格辅助定位根因：

| 读者反馈症状 | 编辑报告对应 | 可能根因 | 追溯到环节 |
|---|---|---|---|
| "中间那段差点划走" | 闷段检测：500字+无冲突 | 场景缺乏冲突设计 | → 编剧 |
| "没什么特别想看下章的" | 章末钩子弱 | 钩子设计不足 | → 架构师蓝图 |
| "爽点不够" | 爽点评分低 | 核心事件戏剧张力不足 | → 架构师蓝图 |
| "主角有点傻" | 角色一致性问题 | 人物行为偏离设定 | → 文体家执行 |
| "看不太懂在讲什么" | 信息密度过高/过低 | 信息流设计问题 | → 编剧 |
| "对话有点假" | 对话质量评分低 | 人物声音区分不足 | → 文体家执行 |
| "开头太慢" | 开篇速度不达标 | 前200字未进入场景 | → 文体家执行 |
| "这章好像没什么进展" | 信息密度低 | 核心事件太弱 | → 架构师蓝图 |
| "读起来像AI写的" | AI味检测🔴/🟡 | 词汇/句式/结构模板化 | → 文体家执行（对照去AI味指南逐项修正） |
| "文字太流畅/太正式" | AI味检测🔴/🟡 | 缺少口语化表达和生活碎片 | → 文体家执行（增加毛边感、短句、语气词） |

## 迭代熔断决策记录

每次触发迭代熔断时，记录到 `outline.md`：

```markdown
### 迭代熔断记录

#### 第N章 — [日期]
- 累计迭代：3次
- 核心问题：[诊断结论]
- 作者决定：[接受/换方向/缩范围/跳过]
- 执行结果：[最终处理方式]
```

## 多本书管理

每本书在 `books/` 下有完全独立的工作空间：

```
books/
├── 都市预知者/          # 书名就是目录名
│   ├── story-state.md
│   ├── characters.md
│   ├── outline.md
│   ├── chapters/
│   └── drafts/
└── 金融猎手/
    ├── story-state.md
    ├── characters.md
    ├── outline.md
    ├── chapters/
    └── drafts/
```

切换书时，只需在命令中指定不同书名。状态文件完全隔离，不会串。
